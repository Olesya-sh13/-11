## 3. Результаты выполнения задания

### 3.1 Оценка процесса менеджмента качества по ГОСТ Р ИСО/МЭК 12207

ГОСТ Р ИСО/МЭК 12207-2010 устанавливает общую структуру процессов жизненного цикла программных средств. Применительно к Security Logger рассматриваются следующие процессы.

#### 3.1.1 Применимые процессы жизненного цикла

**Процессы соглашения:**
- *Процесс приобретения* — определение требований к системе мониторинга безопасности. Для Security Logger требования сформулированы в ТЗ: контроль доступа к виртуальным машинам, обнаружение аномалий, шифрование логов.
- *Процесс поставки* — передача готового программного продукта заказчику через платформу Replit с автоматизированным скриптом развёртывания `setup.sh`.

**Процессы организационного обеспечения:**
- *Процесс менеджмента модели жизненного цикла* — применяется итерационная модель: функциональные требования → реализация → тестирование → развёртывание.
- *Процесс менеджмента инфраструктуры* — использование Python 3.10, Django 5.0.2, SQLite, Gunicorn, библиотек cryptography, pandas, numpy.
- *Процесс менеджмента человеческих ресурсов* — распределение ролей: разработчик (реализация бизнес-логики), администратор БД (миграции), тестировщик (pytest, модульные тесты).

**Процессы технического управления:**
- *Процесс планирования проекта* — определены сроки разработки, список артефактов: исходный код, документация, тесты, скрипт развёртывания.
- *Процесс управления рисками* — идентифицированы риски: компрометация ключа шифрования Fernet, утечка данных в открытых логах, SQL-инъекции. Контрмеры: хранение ключа в `secret.key`, шифрование `.enc`-файлов, использование Django ORM.

**Технические процессы:**
- *Процесс анализа требований* — требования к функциональности: ролевая модель (admin/auditor), мониторинг 4 ВМ, обнаружение частотных аномалий, настройка параметров политики.
- *Процесс проектирования архитектуры* — многоуровневая архитектура MVC: модели (models.py), представления (views.py), шаблоны (templates/), модуль шифрования (encryption.py), модуль симуляции (data/utils.py).
- *Процесс реализации* — разработка на Python/Django, Fernet-шифрование событий безопасности, токен-based аутентификация (обход ограничений cookies в iframe).
- *Процесс тестирования* — модульные тесты (pytest): test_encryption.py, test_models.py, test_views.py.
- *Процесс развёртывания* — автоматизированный скрипт `setup.sh`, Gunicorn на порту 5000.

#### 3.1.2 Политики, стандарты и процедуры менеджмента качества

| Область | Политика | Стандарт | Процедура |
|---------|----------|----------|-----------|

#### Вывод по сравнительному анализу

Для мониторинга качества Security Logger оптимальным выбором является связка **Prometheus + Grafana**:
- Бесплатная и open-source, что соответствует учебному контексту проекта на Replit
- Prometheus собирает метрики HTTP-запросов (количество запросов/сек, коды ответов, время отклика) — напрямую соответствует логике обнаружения аномалий в приложении
- Grafana визуализирует динамику событий, аналогичную встроенному графику «Динамика обращений по времени»
- `prometheus_client` для Python интегрируется в Django за 15 минут

#### 3.3.2 Подходы к качеству

| Подход | Описание | Реализация в Security Logger |

| **Качество продукта** | Соответствие ПО функциональным и нефункциональным требованиям | Ролевая модель доступа; шифрование логов; обнаружение аномалий частоты; настраиваемые параметры политики |

| **Качество данных** | Точность и полнота обрабатываемых данных | Валидация входных параметров (min/max для числовых полей); шифрование критичных данных; целостность policy.json |

#### 3.3.3 Факторы, влияющие на качество

**Внутренние факторы (определяются на этапе разработки):**

- *Архитектурные решения:* Токен-based аутентификация вместо cookies обеспечивает совместимость с iframe-окружением Replit; разделение логики симуляции и UI позволяет независимо изменять алгоритмы без правки шаблонов.
- *Алгоритм обнаружения аномалий:* Скользящее окно 1 секунда с порогами base_frequency и critical_threshold; параметры вынесены в policy.json для гибкой настройки без перекомпиляции.
- *Шифрование:* Fernet (AES-128-CBC + HMAC-SHA256) гарантирует конфиденциальность и целостность логов; ключ хранится в отдельном файле.
- *Покрытие тестами:* Параметризованные тесты шифрования (round-trip), тесты модели пользователя, интеграционные тесты HTTP-представлений.

**Внешние факторы (среда эксплуатации):**

- *Платформа:* Replit (NixOS, Python 3.10, ограниченные ресурсы контейнера) — влияет на производительность при больших объёмах событий.
- *Браузер пользователя:* Iframe-ограничения требуют обходного механизма передачи токена через URL.
- *Gunicorn:* 1 worker process; при высокой нагрузке может стать узким местом.

#### 3.3.7 Взаимосвязь качества программного средства и качества данных

В Security Logger данные и программный код тесно взаимосвязаны:

**policy.json — центральный файл данных:**
- *Точность данных:* Некорректные значения base_frequency или minor_overhead_percent непосредственно влияют на правильность обнаружения аномалий. Решение: валидация на уровне HTML (min/max атрибуты) и в views.py (приведение к int).
- *Целостность данных:* Файл перезаписывается атомарно через `json.dump()` с `indent=2`. При сбое записи — данные сохраняют прежнее состояние.
- *Согласованность данных:* Изменения policy.json немедленно влияют на следующий вызов `generate_events()`, поскольку файл перечитывается при каждом запросе.

**Зашифрованные логи (.enc файлы):**
- *Конфиденциальность данных → защищённость ПС:* Компрометация ключа `secret.key` нарушает как качество данных (читаемость логов посторонними), так и безопасность системы.
- *Целостность данных → надёжность ПС:* HMAC-SHA256 в Fernet обнаруживает любую модификацию зашифрованного файла, гарантируя, что отображаемые администратору данные не были подделаны.

**Симулируемые события (pandas DataFrame):**
- Качество симуляции (coverage нормальных/аномальных сценариев) определяет качество статистики на дашборде. Все 5 типов событий (злоумышленник, нарушения политики, малые аномалии, критические аномалии, нормальные события) покрыты в `generate_events()`.

#### 3.3.8 Приложение. Настройка модели реализации качества с помощью Prometheus

**Выбранная система мониторинга: Prometheus + Grafana**

Обоснование: бесплатная, open-source, нативная интеграция с Python/Django через `prometheus_client`, поддерживает метрики временны́х рядов (запросы/сек), что прямо соответствует логике обнаружения аномалий в Security Logger.

**Шаг 1. Установка зависимостей**

```bash
poetry add prometheus-client
```

**Шаг 2. Создание модуля метрик**

Файл `logger_app/metrics.py`:

```python
"""
Метрики Prometheus для Security Logger.
Экспортирует показатели качества согласно ГОСТ Р ИСО/МЭК 25010.
"""
from prometheus_client import Counter, Histogram, Gauge

# Характеристика: Функциональная пригодность → Функциональная полнота
# Метрика: общее количество проверок политики доступа
POLICY_CHECKS_TOTAL = Counter(
    'security_logger_policy_checks_total',
    'Общее количество проверок политики доступа',
    ['result']  # labels: allowed / denied
)

# Характеристика: Эффективность выполнения → Временна́я эффективность
# Метрика: время обработки запроса к дашборду
REQUEST_DURATION = Histogram(
    'security_logger_request_duration_seconds',
    'Время обработки HTTP-запросов',
    ['endpoint'],
    buckets=[0.1, 0.25, 0.5, 1.0, 2.0, 5.0]
)

# Характеристика: Надёжность → Зрелость
# Метрика: количество аномалий за последний цикл симуляции
ANOMALY_COUNT = Gauge(
    'security_logger_anomaly_count',
    'Количество аномалий в последнем цикле симуляции',
    ['type']  # labels: frequency / critical
)

# Характеристика: Защищённость → Подотчётность
# Метрика: количество изменений политики безопасности
POLICY_CHANGES_TOTAL = Counter(
    'security_logger_policy_changes_total',
    'Количество изменений параметров политики безопасности администратором'
)
```

**Шаг 3. Интеграция метрик в views.py**

```python
@token_required

    REQUEST_DURATION.labels(endpoint='dashboard').observe(time.time() - start)

POLICY_CHANGES_TOTAL.inc()
```
